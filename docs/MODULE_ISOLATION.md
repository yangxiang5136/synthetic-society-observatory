# Module Isolation

The observatory is the platform. External products, assistants, devices, services, policies, and institutions are modules.

Modules are never assumed to be part of the base society.

## Principles

1. The base population starts neutral.
2. Private modules enter only through explicit user instruction.
3. A module is represented as a structured event manifest.
4. The event can be injected, paused, removed, or swapped.
5. Agent memory must not be rewritten to make the module feel historically known.
6. Any branded module must live in a separate manifest, not in the public seed.

## Why This Matters

If a private product is baked into the default world, agent behavior becomes polluted. Everyone starts to act as if they already know the product, its language, its trust cues, and its history.

The platform should test the effect of an event entering a society. It should not silently turn the society into a product-specific universe.

## Manifest Boundary

Each module manifest should answer:

- What is being introduced?
- Is it software, hardware, service, policy, or institution?
- What does it claim to do?
- What are its physical or digital properties?
- Who can see it?
- What can agents know at first contact?
- What must remain unknown unless the event reveals it?
- How can the module be removed without rewriting the population seed?

## Safe Default

The public repository ships only neutral example modules.

Private modules should be added later as separate local or private manifests when a run explicitly needs them.
